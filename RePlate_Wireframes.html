01 Project Overview & Research Findings Context, personas, and key user research from Milestone 1
SECTION 01 / PROJECT OVERVIEW
Why RePlate Exists
Hospital food waste in KSA is largely untracked and fragmented. RePlate bridges the gap between waste generation and certified disposal partners.
P.01

Research Summary (N=15)

⚠️ Core Pain Points

  • Manual tracking is labor-intensive with no standard template
  • Data fragmented across departments, no central view
  • No real-time alerts or threshold monitoring
  • Partner coordination done via phone/WhatsApp
  • No compliance-ready reports for regulators

📊 Survey Results

  • 60% report significant daily food waste
  • 86.7% want real-time digital tracking
  • 73.3% prefer mobile-accessible tools
  • 100% require internal server hosting (MoD)
  • 86.7% need multi-department access

✅ Critical Requirements

  • Minimal manual input — quick logging forms
  • Internal server hosting (Ministry requirement)
  • Role-based access control (4 roles)
  • Response / support SLA: <24 hours
  • CSV / PDF data export for compliance

🏥 System Integration

  • Compatible with existing hospital IT infrastructure
  • Works on hospital intranet (no external cloud)
  • Partner notification via internal messaging
  • Audit trails for all user actions
  • Daily automated backup at 02:00 AM

S-02b · LOGIN (ERROR STATE)

Authentication Notes

1
No Pre-filled Credentials

Login form starts completely empty. Each hospital creates their own accounts in the Admin panel. Zero hardcoded users.

2
Role-first Auth Flow

User picks role before login. This allows role-specific UI rendering and validates credentials against role permissions.

3
Error State Handling

Clear red-bordered fields + inline error message on incorrect credentials. No confusing generic errors.

4
Change Role Link

"← Change role" at the bottom lets users go back without losing context — critical for users who accidentally picked the wrong role.

5
Forgot Password

Directs user to their System Admin rather than email reset — aligns with internal server security policy (no external email servers).

User Account Setup (Admin Panel)

👤 How Hospitals Create Accounts

  • System Admin logs into Admin Panel (S-11)
  • Taps "＋ Add User" button in User Management table
  • Fills in: Name, Email, Role, Department, Status
  • Saves → user appears in the table immediately
  • Admin shares credentials directly with the staff member
  • Staff member logs in with their own email + assigned password

🏢 Partner Organization Setup

  • Admin creates accounts with Role = "Partner"
  • Partner logs in → sees Partner-only screens
  • Cannot access any hospital internal data (waste logs, reports)
  • Only sees pickup requests assigned to their org

🔒 Security Design Principles

  • Zero pre-filled credentials — all fields start empty
  • Role-first authentication flow prevents accidental cross-role access
  • Bottom confirmation dialog before logout prevents accidental sign-out
  • Inactive users cannot log in (controlled from Admin panel)
  • Audit log toggle tracks all user actions for compliance
  • Internal server hosting — no external data transmission (MoD requirement)

📋 Credential Policy (Recommended)

  • Email format: firstname@hospitalname.sa
  • Partners: username@companyname.sa
  • Passwords: minimum 8 characters (enforced at production)
  • Password reset: contact System Admin directly
  • No external email servers used (internal policy)
